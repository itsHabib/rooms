## Goal

A Node script baked into the rootfs that wraps `@cursor/sdk` — same `Agent.create() → send() → stream() → wait()` shape as ship's `LocalCursorRunner`, simplified for one-shot execution inside the microVM. This is what replaces the POC's `claude -p` when a cursor-backed run is requested.

After this lands, `rooms run --runner cursor` (or whatever flag) becomes a real path, and ship's eventual `RoomCursorRunner` (#5) has something to invoke.

## Functional

**Rootfs additions** (amends `scripts/build-rootfs.sh` from #6):
- `nodejs` >= 20.x
- `npm`
- `@cursor/sdk` installed globally OR vendored at `/opt/rooms/cursor-runner/`
- `/opt/rooms/cursor-runner/cursor-runner.js` script (the wrapper)

**`cursor-runner.js`** (Node script, ~150 LOC):

Input: reads from `/workspace/in/`:
- `task.md` — the prompt to send to the agent
- `meta.json` — `{ base_sha, model_id, model_params?, agent_name? }`

Auth: reads `CURSOR_API_KEY` from env. Errors with structured exit code if missing.

Execution (mirrors ship's `LocalCursorRunner.#startAgent` + `#runPipeline`):

```js
const agent = await Agent.create({
  apiKey: process.env.CURSOR_API_KEY,
  model: { id: meta.model_id, params: meta.model_params },
  local: { cwd: "/workspace/repo", settingSources: ["project"] },
  ...(meta.agent_name && { name: meta.agent_name }),
});
const run = await agent.send(taskMd);
for await (const ev of run.stream()) {
  appendEventNdjson(ev);
}
const result = await run.wait();
writeSummary(result);
writeResultJson(result);
await agent[Symbol.asyncDispose]();
```

Output: writes to `/workspace/out/` per the runner contract (#3):
- `events.ndjson` — streamed events, one JSON per line
- `summary.md` — derived from the terminal `RunResult.summary` or final assistant message
- `result.json` — per the runner contract schema
- exits 0 on succeeded, 1 on failed, 2 on SDK error (Agent.create rejection, send rejection, etc.)

`result.patch` is generated by the substrate after the runner exits (substrate runs `git diff base_sha...HEAD` inside the guest); the runner itself doesn't produce the patch.

**Substrate-side change** (`src/runner.rs`):
- Add a `Runner` enum: `Command(Vec<String>)`, `Cursor { model, agent_name? }`, `ClaudeCode` (the POC default).
- `Runner::cursor` shells out to `node /opt/rooms/cursor-runner/cursor-runner.js`, passes `/workspace/in/` populated with `task.md` and `meta.json`.
- Env passthrough: `CURSOR_API_KEY` (mandatory for cursor runner), preserve existing `ANTHROPIC_API_KEY` for claude.

**Error propagation** (matches ship's `LocalCursorRunner` failure modes):
- `Agent.create` rejection → exit 2, writes `result.json` with `status: "failed"` and the error in events.ndjson as a `{ kind: "error", phase: "agent_create", message }` line
- `agent.send` rejection → same, `phase: "send"`
- Stream error before terminal → fall back to `run.wait()`; if that resolves, use it; else exit 2
- `run.wait()` rejection after clean stream → exit 2, `phase: "wait"`
- Successful terminal → exit 0, `status` from `RunResult`

## Tradeoffs

- **Node script vs Rust binary inside the rootfs.** Cursor SDK is TypeScript/Node only. A Rust binary would require maintaining a Rust→Cursor binding; not worth it for v0.1.
- **Mirror ship's LocalCursorRunner shape vs reinvent.** Mirror. The error modes are battle-tested in ship; copying them avoids re-learning the same edge cases (stream-errored-without-terminal, dispose-on-create-failure, etc.).
- **Global npm install vs vendored at `/opt/rooms/cursor-runner/`.** Vendored is reproducible (the same `node_modules` ships in every rootfs); global is one fewer dir. Vendor. Worth the 20MB.
- **Substrate writes result.json vs runner writes it.** Runner writes the `status` field from the SDK's `RunResult`; substrate writes the `timed_out` / `cancelled` overrides if applicable. Two-stage but each side owns what it knows.

## EDs (engineering decisions)

- **ED-1: Vendor `@cursor/sdk` in `/opt/rooms/cursor-runner/node_modules/`.** Reproducibility > disk space.
- **ED-2: Mirror ship's `LocalCursorRunner` error taxonomy (Agent.create / send / stream / wait phases).** Audit by reading `pers/ship/packages/cursor-runner/src/local-runner.ts` and grepping for `CursorRunFailedError`.
- **ED-3: Runner is a separate process from `rooms`-the-binary.** Communicates via filesystem (input dir, output dir), not stdio. Stays compatible with the substrate's "any command" contract.
- **ED-4: Model selection via `meta.json`, not env vars.** Per-room model choice; env vars would force one model per `rooms-host`.
- **ED-5: No streaming runner→substrate while running.** Substrate sees terminal state only; live event tail is available by `tail -f` on `/workspace/out/events.ndjson` if the operator cares.

## Validation

- E2E test (gated `#[cfg(feature = "e2e")]`): create a fixture repo with a `README.md`, write a `task.md` saying "append a line `# rooms` to README.md", run via `rooms run --runner cursor`, assert `result.patch` contains the line addition.
- Auth failure test: run without `CURSOR_API_KEY`; expect non-zero exit, events.ndjson has `{ kind: "error", phase: "agent_create", message: /API key/i }`.
- Stream-without-terminal test: harder to construct; defer to integration with real SDK.

## Risks

- **Cursor SDK auth model changes.** SDK is pre-1.0; auth surface could shift. Mitigation: pin the `@cursor/sdk` version in the vendored install; bump deliberately.
- **Node version drift.** Cursor SDK requires Node >= 20; older Ubuntu base images ship 18. Mitigation: rootfs builder uses NodeSource APT repo to install 20.
- **The runner duplicates ship's `LocalCursorRunner` logic.** Drift between the two is a real risk. Mitigation: when `RoomCursorRunner` (#5) lands in ship, refactor to share — the runner script could even be co-located with ship.
- **Events.ndjson format drift between cursor and claude runners.** Per runner-contract (#3), substrate doesn't parse — but downstream consumers might. Mitigation: when a third runner lands, normalize common keys.

## Out-of-scope

- Cursor cloud runtime support (covered by ship's `CloudCursorRunner`; rooms is local-runtime by definition).
- Mid-run agent steering (`agent.send` twice). v0.1 is one-shot.
- MCP server passthrough into the room (ship supports this; deferred until a real consumer demands it inside a room).
- Streaming events back to the host filesystem mid-run.
