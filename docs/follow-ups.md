Surfaced 2026-06-19 verifying the ship-sudo ripple ([ship#143](https://github.com/itsHabib/ship/pull/143)) on rooms-host — now that ship invokes `sudo -E rooms run`, rooms collects `--out` as root:

- 2026-06-19 — `runner.rs::collect_out_to_host` extracts the guest's `/workspace/out` tar into the host out dir, and host `tar` defaults to `--same-owner` when run as root (the post-#44 sudo path). So collected artifacts land owned by the *archived guest uid* (the guest `rooms` user, uid 1000) rather than the invoking `SUDO_UID`. Harmless on the current host (the `mh` user is also uid 1000), but on a uid-misaligned host the `--out` tree would be owned by uid 1000 ≠ the caller — a non-root consumer (ship's `safeRemove`) then can't clean it up (root-owned `/tmp/ship-rooms-*` leak) and reads can fail under a restrictive umask. Fix: under sudo, chown the collected out dir tree back to `$SUDO_UID:$SUDO_GID` after extraction (or extract `--no-same-owner`). (codex P2 on [ship#143](https://github.com/itsHabib/ship/pull/143))

Surfaced 2026-06-20 by `rooms diff` review ([#47](https://github.com/itsHabib/rooms/pull/47)); the host-side gate exit-code semantics (indeterminate → exit 2) landed in #47, but these touch in-guest enumeration / a bigger re-arch and each wants its own host-e2e:

- 2026-06-20 — **symlink writes aren't enumerated in the upperdir scan.** `ENUMERATE_OVERLAY`'s `find` visits `-type f` (files) + `-type c` (whiteout char-devs) but not `-type l`; a `ln -s target /etc/foo` lands a symlink in the upper that `rooms diff` misses, so a persistent-path lane escape via symlink exits 0. Add `-type l` to the walk (classification falls through the existing add/modify branch). Conservative (can only catch more), but needs a host-e2e since it's in-guest bash. (codex P2 on [#47](https://github.com/itsHabib/rooms/pull/47)) — **Addressed** by [#49](https://github.com/itsHabib/rooms/pull/49): `-type l` added + a `|| [ -L ]` guard for a lower dangling symlink; host-e2e'd a sudo `/etc` escape **and** a non-sudo `/home/rooms` escape → both flagged, exit 3.
- 2026-06-21 — **other special file types aren't enumerated (`mknod` escape).** Even after the symlink fix, `ENUMERATE_OVERLAY`'s `find` walks `-type f`/`-type l`/`-type c` but not `-type b` (block dev) / `-type p` (FIFO) / `-type s` (socket); and a **non-whiteout** char device (a `mknod c` that isn't `0:0`) hits the `-c` branch, fails the `0:0` whiteout test, and is silently dropped. An agent with NOPASSWD sudo could `mknod` a device/FIFO/socket outside `/workspace` and it would be omitted from `changeset.json` → exit 0. Walk `-type b -o -type p -o -type s` and split the `-c` branch (`0:0` → delete, else → add/modify like a file). Same class as the symlink gap; needs a host-e2e (`mknod` a node, confirm flagged). (claude review on [#49](https://github.com/itsHabib/rooms/pull/49)) — **Addressed** by [#50](https://github.com/itsHabib/rooms/pull/50): walk widened to `-type f/l/b/p/s/c`, whiteout check flattened to `[ -c ] && [ 0:0 ]` so a non-`0:0` device falls through to add/modify; host-e2e'd a block dev + non-`0:0` char + FIFO escape (all flagged → exit 3) and a deletion (still classified `D`).
- 2026-06-21 — **directory-only mutations aren't enumerated.** The walk has no `-type d`, so an *empty* dir created outside `/workspace` (`sudo mkdir /etc/evil`) isn't reported (a dir *with* contents is caught via its files); dir mode/owner changes likewise aren't surfaced. Low realism — an empty dir is a weak escape. Walk `-type d` + classify by lower presence, or accept as a known limit. (mapping pass on [#50](https://github.com/itsHabib/rooms/pull/50))
- 2026-06-21 — **ephemeral-prefix payloads evade the lane tripwire.** `is_lane_escape` filters the OS-churn prefixes (`/run`, `/var/log`, `/var/cache`, `/var/tmp`, `/tmp`, ...) to avoid false positives, so a persistent payload an agent drops there (`sudo cp evil /var/log/x`) is in `changeset.json` but **not flagged** as an escape (`--json` still surfaces it for forensics). A filtering-policy tension, not a walk gap: tightening the filter (flag non-OS-owned writes under those prefixes) trades against false-positive noise. (mapping pass on [#50](https://github.com/itsHabib/rooms/pull/50))
- 2026-06-20 — **opaque-directory whiteouts aren't surfaced.** A delete-then-recreate dir carries `trusted.overlay.opaque=y` on the upper dir; the walk skips directories, so the removed entries aren't reported (the recreated dir's new files show as added). Narrow; acknowledged in the rooms-diff spec. Reading the xattr needs `getfattr`/`attr` in the Alpine image + a representation decision (how to express "these lower entries were dropped"). (claude review on [#47](https://github.com/itsHabib/rooms/pull/47))
- 2026-06-20 — **`rooms diff` trusts agent-controllable output (adversarial-guest hole).** The enumeration runs in-guest over SSH as the same NOPASSWD-root `rooms` user the agent runs as, so a compromised guest can forge the `NOOVERLAY` sentinel or omit its own escape records. Exit 0 is therefore a forensic signal against a non-adversarial guest, not a containment boundary (now documented in the spec's *Trust boundary* section). Closing it means enumerating the upperdir **host-side** via the firecracker drive instead of guest SSH — a bigger re-arch, out of scope for #47. (gate-soundness review on [#47](https://github.com/itsHabib/rooms/pull/47))

Surfaced 2026-06-28 by `rooms-registry` (the `rooms ls` + `rooms gc` registry; spec [`docs/features/rooms-registry/spec.md`](features/rooms-registry/spec.md)); deferred to keep the PR a sharp v0:

- 2026-06-28 — **`rooms kill <id>`** (terminate a *live* room) isn't built. An orphaned-but-*alive* firecracker — the launching `rooms run` died but firecracker is still up — shows in `rooms ls` as `running` and is never reaped by gc (gc reaps only confirmed-dead orphans). Reclaiming it needs an explicit `kill` verb (signal the fc pid, then reap). Its own follow-on.
- 2026-06-28 — **gc doesn't free a TAP.** v0 uses one shared `tap-fc0` (every room attaches it; `tap_owned` is always false), so there's nothing per-room to release. When the multi-room pool retires `tap-fc0` for per-room TAPs, `rooms gc` must `ip tuntap del` the orphan's TAP as part of the reap. Pool-era concern; noted, not built.
- 2026-06-28 — **`gc --json`** not implemented. `rooms ls --json` ships (the `doctor`/`diff` stdout contract); `gc` prints a human summary to stderr. A `GcReport` JSON surface is a trivial parity add when a consumer (cloud driver) needs machine-readable reap results.
- 2026-06-28 — **liveness can't see an orphaned-but-alive fc whose launcher renamed it.** `room::probe` keys on `/proc/<pid>/comm ∈ {firecracker, jailer}`; if a future launch path changed the process name, a live room would read `Dead`. The pid is the jailer child that `exec`s firecracker, so this holds today — but a host-side cross-check (the room's `/proc/<pid>/cwd` resolving into its jail root) would make liveness identity-proof if it ever matters.
- 2026-06-28 — **gc can't reap meta-less / pre-feature leaks.** A room with no `room.json` (created before the registry landed, or a crash before the write) classifies `Unknown` and isn't auto-reaped (fail-safe). The rooms-host already carries ~5 such leaked dirs under `~/.local/state/rooms/`. A **socket-probe liveness fallback** (a refused/absent `api.sock` ⇒ definitively dead) would let `gc` clean them; pairs with a **jail-subtree sweep** (`<chroot_base>/firecracker/<id>` with no per-room dir) as a second orphan source. Each is a new liveness/discovery path → wants its own adversarial pass. (rooms-registry spec out-of-scope)
- 2026-06-28 — **adversarial-pass finding (fixed in-PR):** the live teardown (`cleanup_sync`) deleted the per-room dir unconditionally even when a bind-mount couldn't be unmounted (EBUSY at shutdown), stranding the mount where `gc` could never find it. Fixed: `teardown_jail_sync` now reports real success (instance dir gone) and `cleanup_sync` preserves the room dir on failure so the room re-classifies orphaned-dead and `gc` retries. A `umount -l` self-heal in the reap path is a possible further hardening. (rooms-registry adversarial Workflow)
- 2026-06-28 — **`make check` clippy diverges from CI** (recurrence of the #44 cross-OS gap): CI runs `cargo clippy --all-targets --all-features`, but the `Makefile`'s `lint`/`check` omit `--all-features`, so an `e2e`-gated compile error (a stale `boot()` call in `tests/control_failures.rs` after a signature change) passed local `make check` and only failed in CI. Add `--all-features` to the Makefile clippy so local == CI. (rooms-registry, [#51](https://github.com/itsHabib/rooms/pull/51))
- 2026-06-28 — **non-root `rooms ls`/`gc` can't read root-created room state** (codex P2 on [#51](https://github.com/itsHabib/rooms/pull/51)): `sudo -E rooms run` creates the per-room dir 0700-owned by root, so a non-root `rooms ls` gets `PermissionDenied` on `room.json` and the room shows `unknown` (no pid/age), and `gc` skips it. The `/proc` liveness *mechanism* is uid-independent, but the metadata file isn't readable. Today the working path is `sudo -E rooms ls`/`gc` (consistent with `sudo -E rooms run`); to support non-root, chown the room state tree to `$SUDO_UID:$SUDO_GID` after creation (or relax perms). Same class as the 2026-06-19 `--out` chown-under-sudo item.

Surfaced 2026-06-29 by `rooms kill <id>` review ([#53](https://github.com/itsHabib/rooms/pull/53)); the High-severity pre-SIGKILL identity re-probe was fixed in-PR, these two lower-severity kill-path findings were deferred to keep the fix focused:

- 2026-06-29 — **a `reap_paths` failure after a successful terminate exits 2, not 1.** `reap_after_kill` takes `reap_paths(...)?`, so a `PathEscape`/`HomeUnset` from path re-resolution bubbles out of `kill` as a `RegistryError` (CLI exit **2**) instead of a `KillDisposition::Failed` outcome (exit **1**) — the kill/gc contract's code for an incomplete reap. - 2026-05-30 — host-side artifact collection: `rooms run --runner cursor` wrote `events.ndjson`/`summary.md`/`result.json`/`result.patch` into the guest's `/workspace/out`, but the non-`--keep` path tore the VM down before pulling them to the host, so `rooms collect --from` had nothing to read. **Fixed:** `rooms run --out <hostdir>` collects the artifacts back to the host after the run, and `rooms collect --from <hostdir>` reads them. ([#40](https://github.com/itsHabib/rooms/pull/40) `973534b`)

Resolved by `cursor-sdk-runner` (branch `prod-cursor-sdk-runner`; merge SHA on landing):

- 2026-05-29 — `runner.rs` SSHed to the guest as **`root@`**, but the agent rootfs sets `PermitRootLogin no` and runs the agent as the non-root `rooms` user (claude-code refuses `--dangerously-skip-permissions` as root), so `rooms run --command`/`--runner` couldn't drive the Alpine image. **Fixed:** the runner SSHes as **`rooms@`** (a `GUEST_USER` const) and forwards `CURSOR_API_KEY` alongside `ANTHROPIC_API_KEY`. ([#34](https://github.com/itsHabib/rooms/pull/34))
- 2026-05-29 — `runner.rs::seed_entropy` shelled a **python** ioctl over SSH to seed the guest CRNG — a bionic-kernel workaround, now obsolete under the FC CI 6.1.155 virtio-rng kernel (`/dev/hwrng` present) and broken on python-less Alpine. **Fixed:** `seed_entropy` removed (along with the now-dead `RunnerError::EntropySeed`); the `/entropy` device firecracker attaches seeds the CRNG natively. The `/entropy` attach in `firecracker.rs` stays. (`agent-rootfs-alpine-kernel`)
